# Настройка автоматической сборки `.skill` в репозитории SWC Knowledge Base

Этот комплект добавляет в твой репозиторий `SWCSkill/swc-knowledge-base` автоматическую сборку и публикацию `.skill`-файла в GitHub Releases при каждом коммите в `main`.

## Что содержится в архиве `swc-actions-bundle.zip`

```
swc-actions-bundle/
├── .github/
│   └── workflows/
│       └── build-skill.yml          # GitHub Actions workflow
├── scripts/
│   └── build_skill.py               # Скрипт сборки .skill
├── skill-template/
│   └── SKILL.md.template            # Шаблон SKILL.md с переменными
└── README-addon.md                  # Текст для добавления в README
```

## Как установить (5 минут)

### Шаг 1. Залить файлы в репо

Распакуй архив `swc-actions-bundle.zip` локально. Содержимое папки `swc-actions-bundle/` нужно залить в корень репозитория `SWCSkill/swc-knowledge-base`.

Самый простой способ - через веб-интерфейс GitHub:

1. Открой `github.com/SWCSkill/swc-knowledge-base`
2. Нажми **Add file** → **Upload files**
3. Перетащи в браузер содержимое распакованной папки:
   - Папку `.github/` (с workflow внутри)
   - Папку `scripts/` (с `build_skill.py` внутри)
   - Папку `skill-template/` (с шаблоном SKILL.md)
4. В commit message напиши: `Add GitHub Actions for automatic .skill builds`
5. Нажми **Commit changes**

После этого в репо появятся новые папки, а во вкладке **Actions** появится первый запуск workflow.

### Шаг 2. Проверить права на публикацию релизов

Workflow создаёт GitHub Releases, поэтому ему нужны права на запись. По умолчанию в GitHub это работает автоматически для приватных репо и для публичных репо в организациях, но лучше проверить:

1. Открой репо → **Settings** → **Actions** → **General**
2. Прокрути вниз до раздела **Workflow permissions**
3. Убедись, что выбрано **Read and write permissions** (не "Read repository contents permission")
4. Если было выбрано иначе - переключи и нажми **Save**

Это нужно сделать один раз.

### Шаг 3. Обновить файл `VERSION` (опционально, для первого релиза)

Если хочешь, чтобы первый автоматический релиз был с номером `2.2.0`, проверь файл `VERSION` в корне репо:

```
2.2.0
```

Если там `1.0.0` или что-то другое - поправь через веб-интерфейс GitHub (открыл файл → карандаш → пиши `2.2.0` → Commit changes).

### Шаг 4. Обновить README (опционально, для документации)

Открой файл `README.md` в корне репо через веб-интерфейс, нажми карандаш (Edit). В конец добавь содержимое файла `README-addon.md` из этого архива. Закоммить.

### Шаг 5. Первый тестовый запуск

После того как все файлы залиты и настройки проверены:

- **Автоматический вариант:** просто дождись, пока любой коммит в `main` затронет `core/`, `projects/`, `faq/` или `VERSION` - workflow запустится сам
- **Ручной вариант прямо сейчас:**
  1. Зайди в репо → вкладка **Actions**
  2. В левом меню выбери **Build and Release SWC Assistant Skill**
  3. Справа - кнопка **Run workflow**
  4. Нажми её (ветка `main`, поле тега можно оставить пустым)
  5. Дождись завершения (обычно 30-60 секунд)

### Шаг 6. Проверить результат

После успешного запуска:

1. Вкладка **Actions** - увидишь зелёную галочку возле последнего запуска
2. Вкладка **Releases** (справа на главной странице репо) - увидишь новый релиз `v2.2.0` с приложенными файлами
3. Файл `swc-assistant.skill` будет доступен по прямой ссылке:
   ```
   https://github.com/SWCSkill/swc-knowledge-base/releases/latest/download/swc-assistant.skill
   ```

## Как использовать в повседневной работе

### Сценарий 1. Правка FAQ

1. Открой в репо файл `faq/atlas.md` (или любой другой)
2. Нажми карандаш → внеси правку → commit в `main`
3. Через 30-60 секунд в Releases появляется обновлённый `.skill`
4. Отправляешь сотрудникам ссылку `.../releases/latest/download/swc-assistant.skill`

### Сценарий 2. Правка ядра (бренд-гайда)

1. Правишь `core/brand-voice.md` через PR или напрямую
2. Обновляешь `VERSION` (`2.2.0` → `2.2.1`)
3. Коммит → workflow собирает новый релиз
4. Новая версия `v2.2.1` появляется в Releases

### Сценарий 3. Новый проект на витрине

1. Создаёшь новый файл `projects/new-project.md`
2. Добавляешь строку в `projects/INDEX.md`
3. Обновляешь `VERSION` до `2.3.0` (minor bump)
4. Коммит → новый релиз с новым проектом

## Что делать, если workflow упал

1. Открой **Actions** → кликни на упавший запуск (красный крестик)
2. Раскрой упавший шаг - увидишь лог ошибки
3. Самые частые причины:
   - **Permissions** - не сделан Шаг 2 выше (Read and write permissions)
   - **Невалидный YAML** в шаблоне SKILL.md - покажется в логе как ошибка от `yaml.safe_load`
   - **Отсутствующий файл** в `core/` - покажется как `⚠️ Файл не найден`

Если упал непонятно почему - пришли мне скриншот лога, разберёмся.

## Если нужно откатить или удалить

- Удалить любой релиз: **Releases** → нужный релиз → троеточие справа → **Delete**
- Отключить workflow: **Actions** → три точки справа от **Build and Release SWC Assistant Skill** → **Disable workflow**
- Полностью удалить: удали папку `.github/workflows/` из репо
